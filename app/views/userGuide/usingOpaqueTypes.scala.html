Using Opaque Types

Scalactic provides a suite of opaque types that enforce value constraints at compile time, starting with version 3.3.0 build for Scala 3. These types help you prevent invalid states by making it impossible to construct a value that doesn't meet the type's invariant.

In traditional Scala, you might represent a positive integer as Int, but there's nothing stopping someone from passing -5 where a positive value is expected. With opaque types, you can use PosInt to guarantee at compile time that the value is always positive.

Why Opaque Types?

Opaque types provide several benefits:

  • Compile-time safety: Invalid values cannot be constructed, catching bugs before runtime.
  • Self-documenting code: A method signature using PosInt clearly communicates that only positive integers are accepted.
  • Fewer runtime checks: Since the invariant is guaranteed by the type system, you don't need to validate values at runtime.

Numeric Opaque Types

Scalactic provides a comprehensive set of numeric opaque types across Int, Long, Float, and Double. Each type enforces a specific constraint:

ConstraintTypes
Positive (greater than zero) PosInt, PosLong, PosFloat, PosDouble
Non-negative (zero or greater) PosZInt, PosZLong, PosZFloat, PosZDouble
Negative (less than zero) NegInt, NegLong, NegFloat, NegDouble
Non-positive (zero or less) NegZInt, NegZLong, NegZFloat, NegZDouble
Non-zero NonZeroInt, NonZeroLong, NonZeroFloat, NonZeroDouble
Finite (not NaN or infinity) FiniteFloat, FiniteDouble
Finite and non-negative PosZFiniteDouble, PosFiniteDouble
Finite and non-zero FiniteFloat combined with NonZeroFloat

Constructing opaque types

Each opaque type provides an apply method that validates the value at construction time. If the value violates the constraint, a compiler error is raised:

import org.scalactic.opaquetypes.PosInts.PosInt


// Valid construction
val positive: PosInt = PosInt(42)


// This fails with compiler error
val bad: PosInt = PosInt(-5)

Similarly, for non-negative values:

import org.scalactic.opaquetypes.PosInts.PosZInt


val nonNeg: PosZInt = PosZInt(0)    // OK: zero is non-negative

val alsoNonNeg: PosZInt = PosZInt(10) // OK


// This fails with compiler error
val bad: PosZInt = PosZInt(-1)

Character Opaque Types

Scalactic also provides NumericChar, which constrains a Char to be a numeric digit ('0' through '9'):

import org.scalactic.opaquetypes.Numerics.NumericChar


val digit: NumericChar = NumericChar('5') // OK


// This fails with compiler error
val bad: NumericChar = NumericChar('a')

Collection Opaque Types

Non-empty collection types ensure collections always contain at least one element:

  • NonEmptyString — a String that is not empty
  • NonEmptySeq[A] — a Seq with at least one element
  • NonEmptyList[A] — a List with at least one element
  • NonEmptyVector[A] — a Vector with at least one element
  • NonEmptySet[A] — a Set with at least one element

Example with NonEmptyString

import org.scalactic.opaquetypes.NonEmpties.NonEmptyString


val name: NonEmptyString = NonEmptyString("Alice") // OK


// This fails with compiler error
val bad: NonEmptyString = NonEmptyString("")

Using Opaque Types in Tests

Opaque types shine when testing methods that accept constrained values. The type system guarantees correctness, so your tests focus on behavior rather than validation edge cases:

import org.scalatest.flatspec.AnyFlatSpec

import org.scalatest.matchers.should

import org.scalactic.opaquetypes.PosInts.PosInt


class BankAccount {

  def deposit(amount: PosInt): Unit = {

    // amount is guaranteed positive — no runtime check needed

  }

}



class BankAccountSpec extends AnyFlatSpec with should.Matchers {


  val account = new BankAccount()


  it should "accept a positive deposit" in {

    val amount: PosInt = PosInt(100)

    account.deposit(amount)

    // ...assert account balance

  }



  // You don't need a test for negative deposits —

  // the compiler prevents them from being constructed.

}

Where to find the API

All opaque types live in the org.scalactic.opaquetypes package. The key objects include:

  • PosInts — constructors for positive Int-based opaque types
  • PosLongs — constructors for positive Long-based opaque types
  • PosFloats — constructors for positive Float-based opaque types
  • PosDoubles — constructors for positive Double-based opaque types
  • NegInts — constructors for negative Int-based opaque types
  • NegLongs — constructors for negative Long-based opaque types
  • NegFloats — constructors for negative Float-based opaque types
  • NegDoubles — constructors for negative Double-based opaque types
  • NonZeroInts — constructors for non-zero Int-based opaque types
  • NonZeroLongs — constructors for non-zero Long-based opaque types
  • NonZeroFloats — constructors for non-zero Float-based opaque types
  • NonZeroDoubles — constructors for non-zero Double-based opaque types
  • Finites — constructors for finite Float and Double-based opaque types
  • Numerics — constructors for numeric opaque types
